# HU-44 — Nginx: Cabeceras de seguridad HTTP + documentación

## 1. Objetivo

Definir e instrumentar las cabeceras HTTP que reduzcan ataques comunes (XSS, clickjacking, sniffing) y documentar qué hace cada una para que el equipo pueda mantenerlas o ampliarlas. La configuración debe aplicarse sobre el servidor que expone frontend, API y websocket (Nginx) y ser comprobada con herramientas como `curl` y la consola del navegador.

## 2. Cabeceras configuradas en `nginx/default.conf`

Estas cabeceras se activan con `add_header … always` para asegurarse de que incluso las respuestas 4xx/5xx las devuelven.

- `X-Content-Type-Options: nosniff`  
  Impide que el navegador detecte de forma autónoma el tipo MIME de los recursos; protege contra la ejecución de payloads en archivos que contienen código, forzando el uso del tipo declarado.

- `X-Frame-Options: SAMEORIGIN`  
  Evita clickjacking permitiendo que solo la misma página pueda cargar nuestro contenido en un `<iframe>`. Actualmente usamos `SAMEORIGIN` porque algunas rutas administrativas también se cargan desde iframes controlados. Si en el futuro no hay iframes legítimos, se puede cambiar a `DENY` para ser más restrictivos, pero hay que validar que ninguna integración legítima dependa de ello.

- `Referrer-Policy: strict-origin-when-cross-origin`  
  Comparte la URL completa solo con el mismo origen y, en peticiones a otros dominios, transmite únicamente el esquema y host (sin path ni query) en las cabeceras `Referer`. Reduce exposición de datos sensibles en enlaces externos.

- `Permissions-Policy`  
  Se desactivan APIs que el producto no consume, reduciendo la superficie de ataque sobre cámara, micrófono, sensores y pagos. En la configuración actual están negadas: `geolocation`, `microphone`, `camera`, `gyroscope`, `magnetometer`, `accelerometer`, `payment` y `interest-cohort`. Si se añade una característica que necesita alguno de esos permisos, hay que actualizar la cabecera para habilitar explícitamente solo la necesaria.

- `Content-Security-Policy`  
  Política base centrada en `'self'` para scripts, estilos, imágenes, fuentes y conexiones:
   ```
   default-src 'self';
   script-src 'self' 'unsafe-inline';
   style-src 'self' 'unsafe-inline';
   img-src 'self' data:;
   font-src 'self';
   connect-src 'self';
   frame-ancestors 'self';
   base-uri 'self';
   form-action 'self';
   upgrade-insecure-requests;
   ```
   * Las peticiones de API y websocket están cubiertas por `connect-src 'self'`; si se necesita conectar con servicios externos (p. ej. analytics), hay que añadirlos explícitamente.
   * `script-src` necesita `'unsafe-inline'` debido a que el servidor de desarrollo de Vite inyecta scripts inline (importes de módulos y el cliente HMR). Cuando el frontend se compile en producción y deje de usar scripts inline, hay que volver a restringir esta directiva a `'self'` o usar hashes/nonce para mantener la protección contra XSS.
   * `img-src` admite `data:` para iconos en línea que generan algunas librerías, y `font-src` queda en `self` porque todos los assets se sirven desde el mismo origen.
   * `upgrade-insecure-requests` fuerza la carga de recursos por HTTPS y evita que se carguen versiones HTTP en entornos que ya tienen certificados válidos.

  ## 2.1 Bloqueos adicionales en Nginx

  - El bloque `location ~ /\.(?!well-known\/).*` impide servir rutas que empiezan por `.`; eso es útil para ocultar `.git`, `.env`, etc., pero también bloqueaba por accidente los assets de Vite bajo `/.vite/` y causaba los 404 que ves en la consola. Para evitarlo, se agregó un `location ^~ /.vite/` que proxyfía directamente al frontend antes de que llegue el regex de bloqueo.
  - En la misma línea, las rutas de desarrollo realmente se sirven en `/node_modules/.vite/...`, así que también hay que definir un `location ^~ /node_modules/.vite/` que las pase al frontend. Sin este bloque específico los requests caen en el regex general y devuelven 404 aunque el frontend esté arrancado.
  - Si en el futuro se añaden otras rutas generadas por herramientas que usan carpetas ocultas, añade un bloque específico con `location ^~ /ruta/` para pasarlas al frontend antes del filtro general.

## 3. Verificación

1. `curl -I http://<host>` (o `https://` en producción) contra el frontend o la API para ver que las cabeceras aparecen en la respuesta.
2. Navegar por el frontend (home, login, registro, chat) y revisar la consola del navegador; si aparece un bloqueo de CSP, la política deberá ajustar el origen/directiva que lo bloqueó.
3. Probar llamadas a la API (`/api/...`) y la conexión de websocket para asegurarse de que no hay errores de `net::ERR_BLOCKED_BY_CLIENT` o similares.
4. Para módulos nuevos que carguen recursos desde terceros, documentar en este archivo el origen y la directiva que se amplía.

## 4. Mantenimiento

- Si se necesita permitir recursos externos (CDNs, nuevas APIs, sockets), siempre se debe actualizar la línea de `Content-Security-Policy` y justificar el cambio aquí indicando el origen permitido y por qué se confía en él.
- Antes de pasar a producción, confirmar si las rutas con iframes siguen siendo necesarias y decidir si `X-Frame-Options` se mantiene en `SAMEORIGIN` o se endurece a `DENY`; anotar esa decisión en esta sección.
- Para comprobar rápidamente que las cabeceras siguen presentes tras deploys, reutilizar `curl -I` junto con el `grep` de cabeceras o herramientas como `httpie`.
