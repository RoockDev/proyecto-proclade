# 07.2 - Riesgos tecnicos y deuda

## Riesgos prioritarios (P1)

1. Pipeline de staging incompleto
- Impacto: regresiones no detectadas antes de deploy.
- Mitigacion: sustituir placeholders de lint/build/test por ejecucion real.

2. Deuda de lint en backend
- Impacto: reduce fiabilidad de calidad automatica y dificulta refactor seguro.
- Mitigacion: plan de saneamiento por modulo (auth, users, chatbot admin, tests).

3. Upload local sin estrategia de backup
- Impacto: riesgo de perdida de ficheros en recreacion de entorno/servidor.
- Mitigacion: migrar a object storage o implantar backup automatizado de `uploads`.

## Riesgos medios (P2)

1. Falta observabilidad centralizada
- Impacto: diagnostico lento en incidencias reales.
- Mitigacion: logs estructurados + metricas + alertas minimas.

2. Seguridad de borde aun basica
- Impacto: mayor exposicion en produccion real.
- Mitigacion: rate limit, hardening de secretos, auditoria admin.

3. Complejidad creciente del chatbot
- Impacto: mantenimiento dificil sin metricas y gobernanza activa.
- Mitigacion: cerrar HU de admin chatbot y documentar tuning de pesos/umbrales.

## Riesgos bajos (P3)

1. Dependencia de datos seed para demos.
2. Falta de convencion formal de versionado de releases funcionales.

## Plan sugerido de reduccion de deuda

1. Sprint A: cerrar CI staging real + limpiar lint backend critico.
2. Sprint B: observabilidad minima + politicas de backup uploads.
3. Sprint C: automatizacion de pruebas frontend y smoke post-deploy.
