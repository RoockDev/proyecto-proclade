# 07.2 - Riesgos técnicos y deuda

## Riesgos prioritarios

### 1. WebSocket admin del chatbot sin autenticación real

- Impacto:
  - exposición innecesaria del canal admin
- Mitigación:
  - proteger gateway con JWT/rol o token específico de socket

### 2. Pipeline de staging incompleto

- Impacto:
  - se pueden colar regresiones que localmente no se detectan
- Mitigación:
  - sustituir placeholders por lint/build/test reales

### 3. Uploads en filesystem local

- Impacto:
  - pérdida de ficheros si el servidor o el host falla
- Mitigación:
  - backup o migración a object storage

## Riesgos medios

### 4. Hardening de borde todavía básico

- faltan rate limits
- falta política de observabilidad
- el CSP aún permite ciertos recursos externos

### 5. Dependencia de calidad de dato administrativo

- noticias, superhéroes, delegaciones y libros dependen totalmente del panel admin
- un dato mal cargado afecta directamente a la experiencia pública y al chatbot

### 6. Acoplamiento entre chatbot y contenido vivo

- el chatbot se beneficia del contexto dinámico
- también hereda más puntos potenciales de regresión cuando cambian dominios

## Riesgos bajos

### 7. Falta de suite frontend automatizada

- hoy se compensa con lint/build/smoke manual
- a medio plazo limita refactors grandes

### 8. Documentación puede volver a quedarse atrás si no se disciplina el flujo

- esta rama `docs` debe mantenerse viva junto a `dev`

## Recomendación de priorización

1. asegurar WebSocket admin
2. cerrar CI/CD real
3. definir política de backup de uploads
4. introducir tests frontend selectivos
