# 02.5 - Decisiones Arquitectonicas Actuales

## ADR-A001 - Contrato de respuesta unificado

- Decision: toda respuesta backend se normaliza a `{ success, message, data }`.
- Implementacion: `TransformInterceptor` + `GlobalExceptionFilter`.
- Motivo: simplificar consumo frontend y manejo de errores.

## ADR-A002 - Seguridad por JWT + Roles

- Decision: autenticar via Bearer JWT y autorizar con `RolesGuard`.
- Implementacion: `JwtAuthGuard`, `RolesGuard`, decorador `@Roles`.
- Motivo: control simple y mantenible para endpoints de admin.

## ADR-A003 - Soft delete en entidades principales

- Decision: no borrar fisicamente usuarios/noticias en operaciones funcionales.
- Implementacion: campo `deletedAt` y filtros `deletedAt: null`.
- Motivo: trazabilidad y recuperabilidad.

## ADR-A004 - Slug de noticias generado en backend

- Decision: el slug nunca se recibe como dato editable desde frontend.
- Implementacion: `NewsService.generateUniqueSlug`.
- Motivo: consistencia SEO y evitar colisiones/manualidad.

## ADR-A005 - Paginacion de noticias en cliente (estado actual)

- Decision: listado frontend pagina localmente mientras backend no expone metadata de paginacion.
- Implementacion: `buildNewsListState` y `NewsPagination` en frontend.
- Motivo: desbloquear HU de frontend sin bloquearse por backend.

## ADR-A006 - Entorno local unificado con Docker Compose

- Decision: desarrollo principal dentro de contenedores.
- Implementacion: `docker-compose.yml` + Dockerfiles + Nginx.
- Motivo: reducir diferencias de entorno entre miembros del equipo.

## Deuda tecnica abierta relevante

1. CI de staging con jobs `lint/test` aun en placeholder.
2. Redis desplegado pero uso funcional limitado.
3. Falta formalizar contrato OpenAPI/Swagger versionado.
4. Falta estrategia de observabilidad (logs estructurados, metricas, alertas).
