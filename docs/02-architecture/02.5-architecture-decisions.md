# 02.5 - Decisiones arquitectónicas vigentes

## Decisiones activas

1. Contrato HTTP unificado
   - Referencia: `ADR-0001`
2. Seguridad por JWT y roles
   - Referencia: `ADR-0002`
3. Soft delete por defecto
   - Referencia: `ADR-0003`
4. Chatbot híbrido con conocimiento administrable y contexto dinámico
   - Referencia: `ADR-0004`
5. Uploads en filesystem local
   - Referencia: `ADR-0005`
6. Bibliotecas humanas públicas con datos reales
   - Referencia: `ADR-0006`
7. Bootstrap del sistema en arranque Docker
   - Referencia: `ADR-0007`
8. Consentimiento y gateo de servicios externos en cliente
   - Referencia: `ADR-0008`

## Consecuencias directas para mantenimiento

- No se debe asumir que la base de datos se inicializa con usuarios demo.
- El admin inicial depende de `SYSTEM_ADMIN_*` y del primer arranque limpio.
- El frontend depende de `localStorage` para sesión y preferencias de cookies.
- El proyecto usa Nginx como contrato de acceso; probar contra `http://localhost` es más representativo que ir directo al puerto del frontend.
- El chatbot tiene parte HTTP y parte WebSocket; al tocarlo hay que revisar ambos canales.

## Riesgos arquitectónicos visibles

1. El namespace admin del chatbot por WebSocket no aplica autenticación/rol a nivel gateway en el estado actual.
2. Los uploads siguen siendo locales y no tienen backup gestionado por el sistema.
3. El pipeline de staging sigue incompleto y no garantiza lint/test reales.
